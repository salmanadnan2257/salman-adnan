LLM Inference Engine

A from-scratch LLM inference and serving engine in Python and PyTorch, built to understand how systems like vLLM actually work: a paged KV cache with copy-on-write, prefix caching, continuous batching with preemption, speculative decoding, and token streaming behind an OpenAI-compatible API.

Overview

Serving engines are where ML meets systems programming: memory allocators, schedulers, cache coherence, admission control. Reading vLLM's paper is one thing; the goal here was to rebuild the core mechanisms small enough to fit in one head, verify them against a reference implementation, and measure them honestly on real hardware.

Key features

  • An own GPT-2 forward implementation loading Hugging Face weights into plain nn.Linear, verified against transformers to a max abs logit diff under 1e-3 (measured about 3e-5 on distilgpt2).
  • A paged KV cache: fixed-size 16-token blocks, refcounted free lists, per-sequence block tables, and copy-on-write on shared-block writes.
  • Prefix caching: full blocks are content-hashed, so identical prompt prefixes reuse cached blocks with LRU eviction.
  • Continuous batching: requests join and leave the running batch at token boundaries, and when blocks run out the newest sequence is preempted and recomputed later.
  • Speculative decoding: distilgpt2 drafts k tokens, gpt2 verifies them in one forward with accept/reject sampling that preserves the target distribution.
  • A FastAPI server with /v1/completions and SSE streaming, /v1/models, /healthz, and /metrics.

What the numbers say

At equal settings the engine matches Hugging Face one-at-a-time (44.7 vs 40.0 tok/s) and reaches about 2.8x once it can batch 8 requests (113.5 tok/s).

Speculative decoding is correct (identical greedy output, 54% of drafts accepted) but about 0.7x slower on CPU, where the draft forward is not cheap relative to the target. It stays in the codebase as a correctness-verified feature, not a CPU speedup. A comparison against vLLM is deliberately absent because this machine has no GPU.

Challenges

  • Prefix-cache correctness with in-place rewrites: speculative rollback and stop-token truncation rewrite already-hashed positions, so the fix deregisters a block's content hash the moment it is about to be rewritten.
  • Cumulative vs per-sequence capacity checks: per-sequence checks admit a batch that collectively does not fit and crash mid-step, so the check is now summed across the whole batch and pinned by a test.
  • UTF-8 across token boundaries: byte-level BPE can split a character over two tokens, so detokenization holds back a partial tail until it decodes cleanly.

What I learned

  • Paged attention is mostly a bookkeeping problem, not a math problem: the attention kernel barely changes, but the allocator, refcounting, and hashing are where the design lives.
  • Continuous batching's throughput win is real even on CPU (2.5x from concurrency 1 to 8) and comes entirely from not wasting compute on padding.
  • Speculative decoding's speedup is conditional on the draft being much cheaper than the target; measuring it honestly on CPU showed it going backwards, which is a more useful thing to know than assuming GPU numbers transfer.
